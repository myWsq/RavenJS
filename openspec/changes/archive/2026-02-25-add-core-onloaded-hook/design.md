## Context

`modules/core` 当前已有请求生命周期钩子（`onRequest`、`beforeHandle`、`beforeResponse`、`onError`）和插件注册能力，但没有“插件全部加载完成”的统一生命周期。对于需要在插件集合稳定后执行一次性初始化（例如聚合配置校验、跨插件联动预热）的场景，现状只能依赖外部启动代码拼装或手工约定执行顺序。

该改动需要保持 Raven 现有设计原则：单文件核心实现、最小 API 心智负担、与现有异步/错误行为一致，并避免对请求处理热路径引入额外开销。

## Goals / Non-Goals

**Goals:**
- 提供 `onLoaded` 生命周期注册入口，用于在所有插件加载完成后触发。
- 明确 `onLoaded` 的执行时机、调用次数、异步与错误处理语义。
- 与现有生命周期风格保持一致，便于用户理解和测试。

**Non-Goals:**
- 不重构整个插件系统或引入插件依赖图排序机制。
- 不改变已有请求生命周期钩子的触发时机。
- 不在本次变更中引入新的插件卸载或热重载语义。

## Decisions

1. 新增 app-level 生命周期队列而非复用请求钩子队列  
   - 方案：在 `Raven` 内部为 `onLoaded` 单独维护 hook 列表，并提供 `app.onLoaded(hook)` 注册方法。  
   - 原因：`onLoaded` 是应用初始化阶段事件，语义上与 request-scoped hooks 不同。独立队列可避免类型与执行上下文混淆。  
   - 备选：复用 `beforeHandle` 并约定“首次请求前执行一次”。该方案会把初始化语义耦合到请求路径，增加隐式状态和复杂度，放弃。

2. 将触发点放在插件加载流程收尾阶段，并保证仅触发一次  
   - 方案：在“全部插件注册完成”后集中执行 `onLoaded` hooks；重复触发（例如后续再次调用同一启动逻辑）应通过内部状态保护为幂等的一次性执行。  
   - 原因：该时机最符合“插件已就绪”的直觉，且不污染请求热路径。  
   - 备选：每次 `register()` 后都触发。会导致多次运行，不满足“all plugins loaded”语义，放弃。

3. `onLoaded` 支持异步并沿用现有错误传播策略  
   - 方案：按注册顺序 `await` 执行各 hook；若抛错，终止后续 `onLoaded` 并将错误向上传递（由调用方处理）。  
   - 原因：与插件加载阶段 `await`/抛错行为一致，便于用户推理。  
   - 备选：并行执行或吞错继续。并行会引入竞态，吞错会隐藏初始化失败，均放弃。

4. 最小化对外 API 变更  
   - 方案：只新增 `onLoaded` 注册 API（及必要类型），不改变现有 `register` 的返回契约。  
   - 原因：降低破坏性，保持现有插件编写方式稳定。  
   - 备选：引入新的 `bootstrap()` 强制入口。虽然语义更显式，但会扩大迁移面，本次不采用。

## Risks / Trade-offs

- [Risk] 用户在“所有插件完成注册”时机认知不一致（尤其是动态/延迟注册） → Mitigation：在规格中限定触发语义为“当前初始化流程内的插件加载完成后”，并通过测试固化。  
- [Risk] `onLoaded` 抛错导致应用启动失败 → Mitigation：明确这是预期失败路径，错误应显式暴露，避免带病启动。  
- [Risk] 新增一次性状态位可能引入边界 bug（重复初始化、并发调用） → Mitigation：增加并发/重复调用场景测试，确保只执行一次或语义可预测。  
- [Trade-off] 选择串行执行 hooks 会牺牲部分启动速度 → Mitigation：换取确定性执行顺序与更简单调试模型。
