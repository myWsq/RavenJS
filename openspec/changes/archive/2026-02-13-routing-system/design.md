## Context

目前 Raven 框架仅支持单一处理函数，且 Context 组装非常简单。引入路由系统需要解决路径匹配、参数提取以及与现有生命周期钩子（特别是插件作用域下的钩子）的集成问题。

## Goals / Non-Goals

**Goals:**

- 实现基于 Radix Tree 的高效路由匹配。
- 支持路径参数（`:id`）和查询参数的自动解析。
- 支持路由组和嵌套前缀。
- 实现作用域感知的生命周期钩子执行。

**Non-Goals:**

- 提供复杂的正则表达式路由支持。
- 内置复杂的 Body 解析器（保持轻量，由插件处理）。

## Decisions

### 1. 路由算法：Radix Tree

为了平衡匹配速度和参数提取的灵活性，我们将实现一个轻量级的 Radix Tree 路由。

- **理由**: Radix Tree (基数树) 在处理具有共同前缀的路径时非常高效，且能够以 $O(K)$ 的复杂度进行匹配（$K$ 为路径长度）。
- **备选方案**: `path-to-regexp`。虽然简单，但路由数量多时性能较差。

### 2. Context 组装时机

`Context` 的组装将分为两个阶段：

- **阶段 1 (onRequest 前)**: 仅组装基本的 `request`, `url`, `method`, `headers`, `body`。
- **阶段 2 (onRequest 后)**: 执行路由匹配，提取 `params` 和 `query`，完成 `Context` 的完整组装。
- **理由**: 这样可以确保 `onRequest` 钩子能够尽早执行（甚至在路由匹配前），符合“请求到达”的语义。

### 3. 作用域感知的钩子执行 (Hook Pipeline)

为了支持插件/组级别的钩子隔离，我们将采用“编译时”钩子收集策略：

- 当注册一个路由（如 `get`）时，该路由会捕获当前 `Raven` 实例及其所有父实例中已注册的生命周期钩子。
- 匹配到路由后，直接执行这些捕获到的钩子。
- **注意**: 这意味着钩子的注册顺序至关重要（通常应先注册钩子，再注册路由）。

### 4. 路由组实现

`Raven.group(prefix, callback)` 将创建一个临时的 `Raven` 实例，该实例：

- 继承父实例的路由树引用。
- 拥有累加后的 `prefix`。
- 允许在回调中注册组内私有路由。

## Risks / Trade-offs

- **[Risk] 钩子注册顺序** → 路由注册后才添加的钩子可能无法应用到该路由。
  - **Mitigation**: 在文档中明确建议先注册全局/组钩子，再定义路由。
- **[Risk] 内存开销** → 路由树节点较多时会有一定开销。
  - **Mitigation**: 使用高效的对象存储结构，避免重复存储字符串。
